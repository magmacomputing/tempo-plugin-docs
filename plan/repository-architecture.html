Repository Architecture: Monorepos vs. Git Submodules ​

This document outlines the architectural trade-offs between managing the Tempo ecosystem as a unified Monorepo (via NPM Workspaces) versus separating it into independent repositories linked via Git Submodules.

1. Security and Isolation (The Biggest Win) ​

If the primary goal is to allow third-party developers or contractors to work on specific plugins without exposing the proprietary magma core, Git Submodules (or entirely separate repositories) are practically required.

  • The Value: A monorepo is all-or-nothing. If a developer clones magma, they have the whole tree. If you spin tempo-plugin-ticker out into its own repository and link it via a submodule, you can grant granular GitHub RBAC (Role-Based Access Control) to just that plugin.
  • The Verdict: If strict IP isolation is a business requirement for external developers, submodules (or separate repos) will absolutely solve this.

2. Versioning and Release Tracking ​

The ecosystem has varying release cadences, meaning version numbers do not track consistently across plugins.

  • The Submodule Approach: Submodules give each plugin its own isolated Git history, tags, and releases. A plugin can be at v1.2.0 while the core is at v3.0.0.
  • The Monorepo Alternative: You actually do not need submodules to solve independent versioning. Modern monorepo tooling like Changesets or Lerna are specifically designed to handle independent versioning across workspaces in a single Git repository. They automatically track which workspaces changed and bump/tag their versions independently.

3. The "Submodule Tax" (Developer Experience) ​

While submodules solve isolation, they introduce notorious Developer Experience (DX) friction:

  • Detached HEADs: Developers constantly forget to run git submodule update, leading to misaligned commits and build failures.
  • Cross-Boundary Refactoring: In a monorepo, changing an interface in the core instantly highlights breaking changes across all plugins in the IDE. With submodules, cross-repo refactoring becomes a multi-PR chore (update core -> publish/push -> update plugin -> fix -> push).
  • CI/CD Complexity: CI pipelines now have to manage SSH keys and deployment tokens to clone private submodules during the build process.

Recommended Hybrid Strategy ​

Instead of converting the entire monorepo to submodules, a Hybrid Strategy offers the best balance of velocity and security:

  1. Keep First-Party Code in the Monorepo: Keep the magma core, the core licensing engine, and any highly-sensitive proprietary premium plugins in the standard workspace monorepo. Use a tool like Changesets to handle independent versioning. This preserves lightning-fast DX and atomic cross-repo refactoring.
  2. Use Separate Repos for External Developers: When onboarding external developers, create a completely separate, standalone repository for their plugin (e.g., tempo-plugin-community).
  3. Treat Core as a Dependency: External developers shouldn't need the core source code to build a plugin. They should just install the @magmacomputing/tempo package via NPM (or a private registry) as a peerDependency. They build, test, and publish their plugin in total isolation.
