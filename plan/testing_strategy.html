Testing Strategy: Tempo Astro Plugin ​

This plan outlines the verification steps for the @magmacomputing/tempo-plugin-astro package, focusing on mathematical accuracy and licensing integration.

1. Mathematical Accuracy ​

ScenarioTargetExpected Result
J2000 EpochJDE 2451545.02000-01-01 12:00:00 [UTC]
Leap Year HandlingDates around Feb 29Correct day/month rollover without 1-day offsets.
Unix EpochJDE 2440587.51970-01-01 00:00:00 [UTC]

2. Seasonal Calculations (Equinoxes & Solstices) ​

Verify the JDE calculation logic for specific years:

  • Year 2024:
    • Spring: ~March 20
    • Summer: ~June 20
    • Autumn: ~Sept 22
    • Winter: ~Dec 21
  • Negative Years: Verify that calculations remain stable for BCE dates.

3. Licensing Integration ​

ScenarioTest CaseExpected Result
License Injectionnew AstroPlugin({ license: "..." })Plugin initializes without error.
Validation MockMock the validateLicense function.Verify that the plugin correctly passes its permission key ("astro") to the validator.

4. Integration with Tempo Core ​

  • Extension Flow: Tempo.extend(new AstroPlugin()).
  • API Access: Verify tempo.astro.spring() returns a valid Tempo instance.
  • Time Zone Support: Verify that calling .spring({ timeZone: 'Australia/Sydney' }) returns the correct local wall-clock time for the equinox.
