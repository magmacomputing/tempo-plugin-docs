The Open-Core Ghost-Trap Architecture ​

A secure licensing model for JavaScript Monorepos

Introduction ​

Securing intellectual property in JavaScript environments—especially Node.js and frontend frameworks—is notoriously difficult. Because JavaScript is highly dynamic and inherently readable (even when obfuscated), traditional licensing checks like if (license === 'valid') are easily bypassed via memory injection (monkey-patching) or binary patching.

The Open-Core Ghost-Trap architecture is a novel design pattern implemented in the Tempo v3.0.0 ecosystem. It provides enterprise-grade DRM by separating state management into a public, open-source repository while keeping cryptographic validation and access-control within a heavily obfuscated, private closure.

Core Concepts ​

1. The Open-Core Middleman ​

In a standard plugin architecture, the premium plugin both validates the license and manages its own state. In the Ghost-Trap architecture, the Open-Source Engine acts as a middleman.

When the open-source engine initializes, it dynamically imports a proprietary Validator class (if a license key is provided). The Validator performs RSA JWS signature verification and returns a status object. The open-source engine blindly stores this status in an internal, global runtime registry.

typescript
// Open-Source Engine (Public)
const rt = globalThis[Symbol.for('magmacomputing/tempo/runtime')];
rt.license.status = validatorResult.status;

2. Security through Normalcy ​

To a hacker inspecting the open-source codebase, the system looks entirely standard. If they run the software without a license, the public runtime state will reflect a string-based failure:

javascript
console.log(rt.license.status); // Prints: 'invalid' or 'none'

Because the public code uses strings for failure states, an attacker logically assumes the success state is also a string ('active').

3. The Synchronous Signature Handshake ​

Inside the proprietary, obfuscated @core package, the success state is not a generic string like 'active'. It is a mathematically unique checksum derived from the cryptographic signature of the JWT.

typescript
// Proprietary @core Package (Private & Obfuscated)
const SYNC_ANCHOR = "m9fB...k2P"; // Hardcoded secret salt

// The validator extracts the signature, hashes it with the salt, and returns the checksum
const signature = key.substring(key.lastIndexOf('.') + 1);
const checksum = calculateStateChecksum(signature, SYNC_ANCHOR);
return { status: checksum };

When a premium plugin attempts to load via a property getter, the proprietary wrapper independently re-verifies this checksum:

typescript
// Proprietary Plugin Wrapper (Private & Obfuscated)
const s = license.key.substring(license.key.lastIndexOf('.') + 1);
const expectedState = calculateStateChecksum(s, SYNC_ANCHOR);

if (license.status !== expectedState) {
    throw new Error('Premium plugin requires a valid commercial license.');
}

The Attacker's Dilemma ​

Vector A: Memory Injection (Monkey-Patching) ​

The most common attack against JS DRM is injecting the "success" state into the runtime before the plugin loads.

The Trap: The attacker, reading the open-source code, injects the string 'active' into the runtime (rt.license.status = 'active'). When the premium plugin loads, it expects a 32-character hex checksum (e.g., '3cbfcfd6') tied directly to the JWT signature. The string 'active' fails the check.

Vector B: Fake Self-Signed JWTs ​

If the attacker decompiles the plugin, they might notice that the status is verified against the signature of the license.key. They could theoretically generate a fake, self-signed JWT, bypass the RSA validation entirely, and inject their fake signature's checksum into the runtime.

The Trap: To compute the expected checksum to inject into the runtime, they must know the SYNC_ANCHOR (the secret salt). Because the SYNC_ANCHOR is completely obfuscated inside the proprietary bundle and the calculateStateChecksum function is heavily scrambled via AST manipulation, they cannot easily forge the expected string.

Vector C: Source Code Patching ​

The brute-force method requires the attacker to de-obfuscate the proprietary bundle and physically delete the access-control if statements.

The Trap: While technically possible, this requires hours of untangling control-flow flattening. Furthermore, this "crack" is brittle and immediately breaks upon the next software update, forcing the attacker to repeat the grueling process. The Ghost-Trap architecture forces attackers into this high-friction path, heavily disincentivizing piracy.

Conclusion ​

The Open-Core Ghost-Trap is not encryption; it is an architectural hurdle. By misdirecting attackers with open-source string states, and enforcing access via a cryptographically bound, cross-bundle handshake, it makes runtime bypassing virtually impossible, securing your intellectual property without introducing asynchronous performance overhead to legitimate users.
