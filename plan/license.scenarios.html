On this page

  • Customer holds a valid License with 'abc' scope only.

  • Customer calls Tempo.init to activate their AbcTerm.

  • Months later, Customer pays for a new License with an 'xyz' scope only.

In essence, they have two Licenses, each with their own expiry. Their 'abc' is still valid (for a defined period) Their 'xyz' license expires much later.

  • Do they call Tempo.extend to 'add' on their 'xyz' license ? Should they lose access to their 'abc' ? Should we be smart enough (when minting new licenses) to automatically include any of their existing licenses that are not-yet expired ? Their old abc license is still valid and useable, so it would be a shame to deactivate it. It would be a hammer approach to deactivate the old license. They need to be allowed to swap-in their new license on their own schedule.

Antigravity's Thoughts on Scenario 1 ​

1. Should we be smart enough (when minting new licenses) to automatically include existing unexpired scopes? Yes, this is the most secure and robust approach: Server-Side Roll-up Minting. When the customer buys xyz, the backend should mint a single new JWT containing scopes: { abc: {...}, xyz: {...} }. Why?

  • It keeps the client lightweight.
  • It prevents complex client-side merge conflicts (e.g. what if abc is revoked, but the client tries to merge an old abc token with a new xyz token?).
  • The customer only needs to maintain one active license key in their TEMPO_LICENSE_KEY environment variable.

2. Do they call Tempo.extend to 'add' their 'xyz' license? Ideally, yes! They would just call Tempo.extend({ license: 'new-rollup-key' }) to hot-swap the license without restarting the server or losing their custom dynamic configurations. However, note our earlier discovery: We need to patch Tempo.extend first so that it actually triggers the cryptographic validation engine! Right now, extend leaves it in Pending indefinitely.

3. How does Tempo.create fit into this?Tempo.create() creates a local, isolated sandbox for parse configurations (rt.state), but Licensing is strictly a global feature. If you try to pass { license: 'xxx' } into Tempo.create(), the system explicitly rejects it and logs a warning: Licensing is a global-only feature and cannot be set on local instances. Instead, the sandbox inherently inherits the global getRuntime().license. So any license applied via Tempo.init or Tempo.extend automatically trickles down to all sandboxes created via Tempo.create.

Risks of Allowing Sandboxed Licensing ​

While the business cases are strong, decoupling the license from the Global Singleton introduces several critical risks that must be mitigated:

1. Performance & Memory Leaks (The CPU/Network bottleneck) If a multi-tenant SaaS application spins up a Tempo.create({ license: 'tenant-key' }) per request, it will trigger the cryptographic JWS validation pledge every single time.

  • Validating JWTs is CPU-intensive. Doing this thousands of times a second will bottleneck the customer's Node.js server.
  • If the validator checks a remote URL for revocation (e.g., syncRevocation), spinning up sandboxes per-request will effectively accidentally DDoS the Tempo revocation servers.
  • Mitigation: We would need a robust, memory-safe caching layer so that if a Sandbox sees a JWT we've already verified in the last 15 minutes, it skips the expensive crypto/network checks.

2. Plugin Security & The "Ghost-Trap" DRM Currently, premium plugins are secure because they ping getRuntime().license. The getRuntime() function is pinned to a non-configurable, non-writable global slot, making it very hard for a bad actor to spoof the license state.

  • If we move the license into this[$Internal]().license, an attacker has a much smaller surface area to exploit. They might be able to create a fake Sandbox instance or manipulate the prototype chain to inject a mock license object that permanently returns Valid.
  • Mitigation: We would have to ensure the localized state object is deeply frozen and secure against prototype pollution.

3. Context Bleeding and Confusion Plugins are typically registered globally (Tempo.extend(MyPlugin)).

  • If a plugin is registered globally (where the license is valid), but invoked via a Sandbox (where the license is expired/invalid), how does the plugin gracefully handle the rejection at runtime without crashing the customer's app?
  • If a Premium Term registers a custom parsing token, does that token become globally available? If a local Sandbox parses that token without a valid license, does it strip it out or throw an error?

4. Architectural Complexity Moving inter-module state to TempoRuntime was done to unify and harden the system. Splitting the license back out into localized states introduces edge cases where the core engine and the plugins might disagree on what the current active license actually is.
